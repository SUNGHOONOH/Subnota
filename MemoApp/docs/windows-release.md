# Windows Release Notes

Current status: Windows release packaging is not initialized yet.

The existing desktop target is `macos/` via React Native macOS. Windows requires a separate `windows/` native project generated by React Native for Windows, and it should not be generated until the React Native and React Native for Windows versions are aligned.

## Current Blocker

This app currently uses:

```text
react-native: 0.85.2
react-native-macos: 0.81.7
```

React Native for Windows stable releases are version-aligned with React Native major/minor versions. Do not add `windows/` until there is a compatible `react-native-windows` line for the app's React Native version, or until the app intentionally moves to a supported aligned version.

## Safe Preparation

Run:

```sh
corepack pnpm windows:check
```

Expected current result:

- Windows host is required for packaging.
- `windows/` is not initialized.
- `react-native-windows` is not installed.
- React Native and React Native Windows versions are not aligned.

## When Versions Are Aligned

On a Windows machine with Visual Studio Build Tools installed:

```sh
corepack pnpm add react-native-windows@<matching-version>
corepack pnpm exec react-native init-windows
corepack pnpm exec react-native run-windows
```

Before committing the generated `windows/` folder:

- keep existing iOS and macOS projects unchanged,
- review `metro.config.js` because the Windows initializer can overwrite it,
- verify navigation, gesture, safe-area, SVG, Supabase auth, and AsyncStorage behavior on Windows,
- confirm `MEMO_BACKEND_URL` is HTTPS for release builds,
- create Release/MSIX package from Visual Studio after the app launches without Metro.

## Release Target

Preferred Windows artifact:

- MSIX for signed Windows distribution.

Fallback for private testing:

- unpackaged Release build from Visual Studio, only for internal validation.
